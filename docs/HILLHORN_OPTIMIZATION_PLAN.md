# План оптимизации Hillhorn (без снижения эффективности)

Принцип: любое изменение не должно ухудшать качество поиска, полноту контекста или точность recall.

---

## Фаза 1: Принятие агентами (usage, без изменения кода памяти)

| # | Задача | Эффект | Риск для эффективности |
|---|--------|--------|------------------------|
| 1.1 | Дополнить .cursorrules — напоминание о HILLHORN_MODE и первом вызове get_context | Больше вызовов памяти | Нет |
| 1.2 | Создать AGENTS.md — явный workflow для Cursor Agents | Агенты чаще следуют порядку | Нет |
| 1.3 | Усилить MCP tool descriptions — «вызывать в начале сессии» | Рост использования | Нет |

Результат: больше агентов используют Hillhorn, эффективность не меняется.

---

## Фаза 2: Оптимизация без потери качества

| # | Задача | Описание | Сохранение эффективности |
|---|--------|----------|--------------------------|
| 2.1 | Ленивая загрузка NWF Field | Загружать Field при первом search/add, кешировать в модуле. Первый вызов как сейчас (~2s), последующие быстрее | Результаты поиска идентичны |
| 2.2 | Сессионный кеш search | Кеш (query, project_id, k) -> results. Инвалидация при add_to_memory в том же project_id | При add память обновляется — кеш сбрасывается, новые данные в следующий search |
| 2.3 | HTTP-пул для Gateway | Использовать один httpx.AsyncClient для Gateway | Латентность API та же, меньше накладных расходов |

Условия:
- Кеш TTL не более сессии (или до add)
- Никакого уменьшения k, длины сниппетов, отключения agreement_ratio/potential

## Фаза 4: Что НЕ делать (защита эффективности)

| Запрет | Причина |
|--------|---------|
| Не уменьшать k в search | Снижение recall |
| Не обрезать сниппеты ниже 200 символов | Потеря контекста |
| Не отключать agreement_ratio, potential | Хуже калибровка |
| Не кешировать дольше сессии без инвалидации при add | Устаревшие данные |
| Не менять Махаланобис на L2 | Уже реализовано, L2 хуже по статье |
| Не удалять search_similar из merge | Меньше источников контекста |

## Критерии приёмки

- search_memory: те же результаты при том же запросе до/после оптимизации.
- add_to_memory: новые записи сразу видны в search (без лишней задержки).
- hillhorn_get_context: объём и качество контекста не снижаются.
- Тест run_hillhorn_snake_test: время local-шагов не растёт; поиск находит новые add.
